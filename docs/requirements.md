# BBQお買い物サポートアプリ 要件書

- 作成日: 2026-07-15
- 版数: v0.1（ドラフト）

## 1. 背景・目的

BBQ・キャンプの幹事や買出し担当は、参加人数（大人・子供）に応じて品目ごとの必要量や予算を毎回手計算しており、負担が大きい。また経験と勘に頼った分量決定は、過不足（フードロス／買い忘れ）を招きやすい。

本アプリは、過去の実績データ（`5月キャンプ標準買い出しリスト.xlsx` 等）をもとにした原単位（大人・子供1人あたりの必要量）から買い物リストを自動生成し、幹事・買出し担当の検討負担を減らすとともに、適正量の提示によりフードロスを削減することを目的とする。

## 2. 想定利用シーン・規模

- 個人〜小グループ（BBQ・キャンプの幹事1名＋買出し担当数名）が、1イベントにつき1つの買い物リストを作成・共有する。
- 同時アクセスは1リストあたり数名〜数十名程度を想定する（参考データでは大人38名・子供16名規模のイベント例あり）。
- 大規模同時多発アクセス（不特定多数が同時に多数のリストを操作するケース）は本バージョンのスコープ外とする。

## 3. 用語定義

| 用語 | 説明 |
|---|---|
| 原単位 | 大人・子供1人あたりの品目別の基準必要量（例：牛肉 大人135g／子供100g） |
| 共有コード | リストごとに発行される8桁コード。第三者がこのコードを知っていればリストの閲覧・編集が可能 |
| チェック状態 | 品目ごとの「購入済み／未購入」の状態 |

## 4. 機能要件

### 4.1 買い物リスト生成機能

- **FR-1.1** ユーザーは、BBQ参加人数（大人・子供）を入力し、原単位データに基づいて品目別の必要量・購入数量・予算（小計／合計）を自動計算した買い物リストを生成できる。
  - 参考データの列構成（カテゴリ、品目、大人/子供の基準数量、単位、購入数量、単価、小計、備考）を踏襲する。
  - カテゴリ例：肉、野菜(焼き用)、焼きそば系、おつまみ、アヒージョ、おやつ、朝食・おやつ、雑貨・調味料、飲み物、100円雑貨、燃料 等。
- **FR-1.2** ユーザーは、生成されたリストの各アイテムについて、数量の増減・アイテムの追加・削除を行える。
- **FR-1.3** リスト生成後も、いつでも上記の増減・追加・削除の変更が可能（買出し当日まで随時編集可）。
- **FR-1.4** ユーザーは、OpenAI APIを用いた自然言語指示（例：「子供を3人増やして」「ビールを減らして代わりにノンアルを増やして」）によって、リストの内容を修正できる。
  - AIによる修正結果は即座にリストへ反映し、ユーザーが結果を確認できること。
  - AIの誤った解釈によるリスト破壊を防ぐため、修正前後の差分が分かる形で提示することが望ましい。
  - **FR-1.5** AIによるリスト修正指示は、1リストあたり累計30回までとする。上限に達した場合、ユーザーにその旨を通知し、以降のAI呼び出し（自然言語指示）を受け付けない（通常の手動編集は上限後も可能）。

### 4.2 購入状態の共有機能

- **FR-2.1** リスト生成時、システムは8桁の共有コードを自動発行する。
- **FR-2.2** 他のユーザーは、共有コードを入力するだけで同一リストにアクセスできる。O-AUTHログインやパスワード入力は不要とする。
  - 共有コードは実質的なアクセスキーを兼ねるため、第三者による総当たり推測への耐性を考慮した発行・検証方式とする（詳細は非機能要件 NFR-2 を参照）。
- **FR-2.3** ユーザーは、スマートフォンから簡単な操作（タップ等）で各品目の購入チェックを行い、購入状況を共有できる。
- **FR-2.4** あるユーザーがチェックを入れる／外すと、同じリストを開いている他ユーザーの画面表示に、1秒以内を目安に反映される（リアルタイム同期）。
- **FR-2.5** チェックを入れる／外す操作時には、操作にあわせた効果音（SE）を再生する。
- **FR-2.6** チェックを外す操作を行う際は、誤操作防止のため確認ダイアログを表示し、ユーザーの明示的な確認を経てから解除する。

### 4.3 フィードバック機能

- **FR-3.1** システムは、買い物リストの品目のうち50%以上が「購入済み」としてチェックされた日の19時以降に、アンケートを表示する。表示は1リストにつき1回限りとし、一度表示（条件成立）した後は、回答有無に関わらず再表示しない。
- **FR-3.2** アンケートは以下の項目を含む。
  - 大きく過不足が生じた品目と、その原因として思い当たること（自由記述／選択式の併用を想定）
  - 他に欲しかったもの（自由記述）
- **FR-3.3** 収集したフィードバックは、将来的な原単位データの精度改善に活用できるよう保存する（本バージョンでは収集・保存までをスコープとし、原単位への自動反映は対象外）。

## 5. 非機能要件

- **NFR-1（データ永続化）** ユーザーデータ（リスト内容、チェック状態、共有コード、フィードバック回答等）は、DB等の永続化可能なストレージに保存する。
- **NFR-2（データ・ロジックの非公開化）** 本リポジトリはGitHub上でpublic公開する方針のため、以下は絶対にソースコード・リポジトリ内に平文で含めない。
  - 買い物リストの原単位となる実績データ、および人数・条件に応じた補正ロジックの詳細（係数・計算式そのもの）
  - OpenAI APIキー等の秘匿情報
  - 対応方針：原単位となる基準データは `local/data/` 配下に置き、この配下は暗号化しない限りGitHubにアップロードしない（`.gitignore` でGit管理対象外とする）。暗号化していないデータをコミット・pushしないことを徹底する。アプリの実行時参照はDB側（本番データ）を正とし、`local/data/` はローカル開発・検証用の位置づけとする。
  - リポジトリには原単位データそのものではなく、サンプル／ダミーデータのみを含める。
  - APIキー等は環境変数として管理し、`.env` 等はGit管理対象外（`.gitignore`）とする。
  - 共有コードについても、推測されにくい生成方式（十分なランダム性）とし、リポジトリ内にコード生成アルゴリズムの弱点となる情報を残さない。
- **NFR-3（UI/UX）** レイアウトはスマートフォンでの表示・操作を最優先に設計する（モバイルファースト）。PC等での閲覧は表示崩れが起きない程度の対応で可とする。
- **NFR-4（リアルタイム性）** チェック状態の共有は1秒以内の反映を目標とする（FR-2.4参照）。
- **NFR-5（デザイン）** 骨付き肉モチーフのかわいいキャラクターをUI上に配置し、親しみやすいデザインとする。
- **NFR-6（想定規模）** 同時アクセスは1リストあたり数十人規模を想定した設計とし、過度な大規模分散設計は行わない。
- **NFR-7（編集競合）** リスト・チェック状態の同時編集に対する競合制御は行わず、Last Write Wins（後から書き込まれた内容が有効）とする。
- **NFR-8（データ保持）** リスト・共有コードに有効期限は設けず、無期限に保持する（明示的な削除機能は本バージョンではスコープ外）。

## 6. システム構成（想定）

| レイヤ | 技術（想定） |
|---|---|
| フロントエンド | Next.js（React）、スマホ表示を優先したレスポンシブUI |
| バックエンド | Next.js API Routes、または軽量なAPIサーバー |
| データベース | Supabase等のマネージドDB（リアルタイム購読機能を活用し、FR-2.4のリアルタイム反映を実現） |
| AI連携 | OpenAI API（自然言語によるリスト修正指示の解釈） |
| 認証 | 不要（共有コード方式。FR-2.2） |

※ 上記は初期方針であり、詳細設計フェーズで見直す可能性がある。

## 7. データモデル（概要・案）

- **List（買い物リスト）**：id、共有コード（8桁）、大人人数、子供人数、作成日時、アンケート表示条件の判定用フラグ 等
- **Item（品目）**：list_id、カテゴリ、品目名、基準数量（大人／子供）、単位、購入数量、単価、小計、備考、購入チェック状態、更新日時
- **BaseData（原単位マスタ・非公開）**：カテゴリ、品目、大人原単位、子供原単位、単価、備考 ── DB内で管理する。ローカル開発用の元データは `local/data/` に置き、暗号化しない状態ではGitHubにアップロードしない（`.gitignore` 対象）
- **Feedback（アンケート回答）**：list_id、回答日時、過不足品目と原因、他に欲しかったもの

## 8. スコープ外（本バージョンでは対応しない）

- O-AUTH等によるユーザー認証・アカウント管理
- 原単位データへのフィードバックの自動反映（機械学習的な自動補正）
- 大規模同時アクセス（数百〜数千人規模）を前提としたインフラ設計
- 決済・注文（実店舗ECとの連携）機能

## 9. 確定事項（運用ルール）

- **リスト・共有コードの有効期限**：無期限。イベント終了後もデータを削除せず保持する。
- **リスト編集の競合制御**：楽観ロック等は設けず、実装しやすい方式（Last Write Wins：後勝ち）とする。同時編集による競合の検知・マージ処理は行わない。
- **フィードバックアンケートの表示**：1リストにつき1回限り表示する（FR-3.1の条件を満たした最初のタイミングで表示し、回答有無に関わらず以降は再表示しない）。
- **OpenAI APIの利用コスト管理**：1リスト（1案件）あたり、自然言語によるリスト修正指示の呼び出しは30回までとする。上限に達した場合はその旨をユーザーに通知し、以降のAI呼び出しを拒否する。
